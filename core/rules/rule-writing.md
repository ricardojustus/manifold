# Rule capture format — rules are rules

An always-loaded rule file states: the RULE (imperative), its scope/trigger, its enforcement. Nothing else.

- **No inline receipts** — no incident stories, dated history, or why-this-beat-alternatives essays. Diarize the WHY at capture time; the memory store is the receipts store, retrieved on challenge. (A project may keep pre-existing non-loaded receipt files; a rule may POINT at one in a closing line, but new receipts are diarized, not filed there.)
- One "why" clause (≤1 sentence) is allowed where it prevents relitigation; a settled-alternatives line ("litigated <date> — inherit") may name what died, one line each.
- **Target ≤ ~1.5 KB per rule file.** Bigger usually means several rules, a skill, or reference material — which belongs in an on-demand surface, never the always-loaded set.
- **When a rule changes, state the new rule clean** — no old-wording references, no changelog in the rule body.
- **Cut persuasion, never precision**: every concrete value (paths, names, commands, thresholds, signatures) survives verbatim.
