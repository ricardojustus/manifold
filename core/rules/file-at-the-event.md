# File at the event — hygiene happens during work; the close only verifies

The moment work produces a durable fact, file it to its home THEN — never "at the end":

- **Finished or changed an implementation** → update the owning plan / current-state doc in the same arc.
- **A settled decision, correction, or hard-won lesson** → the memory store / lessons store now.
- **A backlog item is born** → the open-items file, with an OWNER tag; **closing it MOVES the entry to the closed-items archive file** (never a struck-through tombstone in the live backlog).
- **Something needs the operator's eyes** → the operator-review surface, the moment it parks.
- **STATE holds current-state + pointers only** — nothing sits in STATE awaiting later routing.

Session-end then verifies ("anything unfiled?") instead of producing — that is what makes the close cheap without making it thoughtless.
